Authentication

Difficulty: Beginner · Estimated time: 15 min

Before managing any OpenStack resource you need a valid, authenticated connection. The Keystone identity service handles authentication and issues tokens that authorize subsequent API calls. In this course we use application credentials — a safer alternative to user credentials that can be scoped and revoked independently.

1. What Are Application Credentials?

Application credentials are project-scoped secrets created from your OpenStack user account. They let scripts and automation authenticate without exposing your main username and password. Key benefits:

  • Can be restricted to specific roles and access rules
  • Revocable at any time without changing your user password
  • Ideal for CI/CD pipelines, lab environments, and automation

2. Creating a Keycloak Client for API Access

In Keycloak, Clients represent applications and services that can request authentication of a user. To allow an external application (script, CI/CD pipeline, microservice) to authenticate against OpenStack through Keycloak, you need to register it as a client.

Step 1 — Open the Clients section

Log in to the Keycloak administration console, select your realm, then navigate to Clients in the left menu and click Create client.

Step 2 — Configure the client

Fill in the following fields:

  • Client ID — a unique identifier for your application (e.g. openstack-lab-client).
  • Client Protocol — select openid-connect.
  • Root URL — the base URL of your application (optional for CLI-based clients).

Step 3 — Set the access type

On the Settings tab, configure the access type depending on your use case:

  • Confidential — for server-side applications that can securely store a secret. A Client Secret is generated in the Credentials tab.
  • Public — for browser-based or mobile applications that cannot keep a secret.

Step 4 — Retrieve the client credentials

For confidential clients, go to the Credentials tab to copy the Client Secret. Use the Client ID and Secret to request tokens from Keycloak's token endpoint: 

# Request a token using client credentials
curl -s -X POST "https://keycloak.example.com/realms/your-realm/protocol/openid-connect/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=client_credentials" \
  -d "client_id=openstack-lab-client" \
  -d "client_secret=YOUR_CLIENT_SECRET" | python3 -m json.tool

3. Creating Keystone Application Credentials (optional)

This step is optional. Keystone application credentials are project-scoped secrets managed directly by the OpenStack Keystone identity service. They provide an alternative authentication method that does not go through Keycloak.

To create Keystone application credentials, authenticate to the OpenStack Horizon dashboard (or use the CLI) with your user account. Then navigate to Identity → Application Credentials and click Create Application Credential.

Fill in the following fields:

  • Name — a descriptive name for this credential (e.g. lab-credential).
  • Description — optional, helps identify the purpose.
  • Expiration Date — optional, set an expiry for extra security.
  • Roles — leave empty to inherit all roles from the user, or select specific roles.

After creation, the ID and Secret are displayed once. Copy them immediately — the secret cannot be retrieved later.

You can also create Keystone application credentials via the CLI:

# Create a Keystone application credential
openstack application credential create my-lab-credential

# The output contains the ID and Secret — save them securely

4. Environment Variables

Export your application credential values so the lab framework can authenticate automatically.

export OS_AUTH_URL="https://auth.cloud.ovh.net/v3"
export OS_AUTH_TYPE="v3applicationcredential"
export OS_APPLICATION_CREDENTIAL_ID="${OS_APPLICATION_CREDENTIAL_ID}"
export OS_APPLICATION_CREDENTIAL_SECRET="${OS_APPLICATION_CREDENTIAL_SECRET}"
export OS_REGION_NAME="GRA7"

5. Exercise 1 — Authenticate with Application Credentials

Use the openstack CLI to authenticate against Keystone using your application credential and obtain a token. 

# Request a token using application credentials
openstack token issue

# The command outputs a token ID and expiration date
# The CLI uses the OS_* environment variables automatically

6. Exercise 2 — Token Management

Learn to validate and revoke authentication tokens. This is essential for understanding token lifecycle in long-running automation. 

# Issue a token and capture its ID
TOKEN=$(openstack token issue -f value -c id)
echo "Token obtained: $TOKEN"

# Revoke a token you no longer need
openstack token revoke "$TOKEN"
echo "Token revoked successfully"

7. Error Handling

Authentication failures are reported with clear error messages by the CLI. If your credentials are wrong, the CLI will return an error. 

# Example: attempting authentication with wrong credentials
# will produce an error like:
#   The request you have made requires authentication. (HTTP 401)

# Always verify your environment variables are set correctly
env | grep OS_

Summary

  • Use application credentials instead of user credentials for safer automation.
  • Set OS_AUTH_TYPE=v3applicationcredential along with your credential ID and secret.
  • Use openstack token issue to obtain a token and openstack token revoke to revoke it.
  • The CLI reads OS_* environment variables automatically for authentication.
  • Authentication errors are reported clearly by the CLI.
