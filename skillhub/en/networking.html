Networking

Difficulty: Intermediate · Estimated time: 20 min

Networks are the foundation of any cloud deployment. In this lesson you will create virtual networks, configure subnets with CIDR validation, and set up routers — all using the Neutron API through the lab framework.

1. Exercise 1 — Create a Network

Use the openstack CLI to create a virtual network named {student_id}-net-network. 

# Create a virtual network
openstack network create {student_id}-net-network

# Verify the network was created
openstack network show {student_id}-net-network -f value -c status

2. Exercise 2 — Create a Subnet

Create a subnet within your network. Use 10.0.0.0/24 as the address range. 

# Create a subnet with CIDR 10.0.0.0/24
openstack subnet create \
  --network {student_id}-net-network \
  --subnet-range 10.0.0.0/24 \
  {student_id}-net-subnet

# Verify the subnet was created
openstack subnet show {student_id}-net-subnet

3. VLAN-backed Networks and Ports

In production environments, networks are often backed by specific VLANs. You can create a VLAN-backed network and assign a port to it so that traffic is tagged with the correct VLAN ID on the physical switch.

Create a VLAN-backed network

# Create a network with a specific VLAN segmentation ID
openstack network create \
  --provider-network-type vlan \
  --provider-physical-network physnet1 \
  --provider-segment 100 \
  {student_id}-vlan100-network

# Create a subnet on the VLAN network
openstack subnet create \
  --network {student_id}-vlan100-network \
  --subnet-range 10.100.0.0/24 \
  {student_id}-vlan100-subnet

Create a port on a specific VLAN network

# Create a port bound to the VLAN network
openstack port create \
  --network {student_id}-vlan100-network \
  {student_id}-vlan100-port

# Verify the port details (VLAN binding)
openstack port show {student_id}-vlan100-port

4. Baremetal VLAN Assignment

When working with baremetal instances (Ironic), you can assign VLAN networks directly to physical interfaces. This is useful for isolating traffic on dedicated hardware. 

# Create a VLAN network for baremetal use
openstack network create \
  --provider-network-type vlan \
  --provider-physical-network physnet1 \
  --provider-segment 200 \
  {student_id}-bm-vlan200

# Create a subnet
openstack subnet create \
  --network {student_id}-bm-vlan200 \
  --subnet-range 10.200.0.0/24 \
  {student_id}-bm-vlan200-subnet

# Create a port for the baremetal node
openstack port create \
  --network {student_id}-bm-vlan200 \
  --vnic-type baremetal \
  {student_id}-bm-port

# Deploy a baremetal instance using this port
openstack server create \
  --flavor baremetal \
  --image "Ubuntu 22.04" \
  --port {student_id}-bm-port \
  --wait \
  {student_id}-bm-instance

5. Trunk Ports

Trunk ports allow a single parent port to carry traffic from multiple VLAN networks via subports. This is commonly used when an instance needs access to several VLANs simultaneously. 

# Create the parent port
openstack port create \
  --network {student_id}-net-network \
  {student_id}-trunk-parent

# Create a subport on a different VLAN network
openstack port create \
  --network {student_id}-vlan100-network \
  {student_id}-trunk-sub1

# Create the trunk with the parent port and attach the subport
openstack network trunk create \
  --parent-port {student_id}-trunk-parent \
  --subport port={student_id}-trunk-sub1,segmentation-type=vlan,segmentation-id=100 \
  {student_id}-trunk

# Add another subport later
openstack port create \
  --network {student_id}-bm-vlan200 \
  {student_id}-trunk-sub2

openstack network trunk set \
  --subport port={student_id}-trunk-sub2,segmentation-type=vlan,segmentation-id=200 \
  {student_id}-trunk

# Verify the trunk configuration
openstack network trunk show {student_id}-trunk

6. QinQ (802.1ad) — Double VLAN Tagging

QinQ (also known as VLAN stacking or 802.1ad) encapsulates one VLAN tag inside another. This is useful when a service provider needs to transport customer VLAN traffic across its own VLAN infrastructure without conflicts. The outer tag (S-VLAN) identifies the provider segment, while the inner tag (C-VLAN) preserves the customer's original VLAN.

Use case

A hosting provider runs multiple tenants on the same physical infrastructure. Each tenant uses their own VLANs internally (e.g. VLAN 10, 20, 30). With QinQ, the provider wraps each tenant's traffic in a unique outer VLAN so that tenant VLAN IDs can overlap without interference. 

# Create a provider network with VLAN type for the outer (S-VLAN) tag
openstack network create \
  --provider-network-type vlan \
  --provider-physical-network physnet1 \
  --provider-segment 500 \
  {student_id}-qinq-provider

# Create a subnet on the provider network
openstack subnet create \
  --network {student_id}-qinq-provider \
  --subnet-range 10.50.0.0/24 \
  {student_id}-qinq-provider-subnet

# Create the parent port on the provider network
openstack port create \
  --network {student_id}-qinq-provider \
  {student_id}-qinq-parent

# Create a trunk and attach a subport with the inner (C-VLAN) tag
openstack network trunk create \
  --parent-port {student_id}-qinq-parent \
  --subport port={student_id}-trunk-sub1,segmentation-type=vlan,segmentation-id=10 \
  {student_id}-qinq-trunk

# The instance sees the inner VLAN 10 traffic,
# while the physical network carries it inside outer VLAN 500
openstack network trunk show {student_id}-qinq-trunk

Summary

  • Use openstack network and openstack subnet commands.
  • Subnets require a valid CIDR range (e.g. 10.0.0.0/24).
  • VLAN-backed networks use --provider-network-type vlan with a --provider-segment ID.
  • Baremetal ports use --vnic-type baremetal to bind to physical interfaces.
  • Trunk ports carry multiple VLANs via openstack network trunk commands.
  • QinQ (802.1ad) stacks an inner C-VLAN inside an outer S-VLAN for multi-tenant isolation.
