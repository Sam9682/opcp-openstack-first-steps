First Steps

Difficulty: Beginner · Estimated time: 20 min

Before diving into individual services, let's review the foundational concepts of OpenStack and discover the different ways to interact with the platform to create networks, instances, volumes, and more.

1. Core Concepts

Projects

A project (also called a tenant) is an organizational unit in OpenStack. Resources such as instances, networks, and volumes belong to a project.

Services

OpenStack is composed of independent services:

  • Nova — Compute (virtual machine instances)
  • Neutron — Networking (virtual networks, subnets, routers)
  • Cinder — Block Storage (volumes, snapshots)
  • Keystone — Identity (authentication, authorization)
  • Glance — Image (virtual machine images)

Endpoints

Each service exposes HTTP endpoints listed in the Keystone service catalog. Clients discover service URLs by querying the catalog after authentication.

2. Accessing the OpenStack Environment

OpenStack provides several ways to manage your infrastructure. Depending on your use case — quick manual operations, scripting, or full infrastructure-as-code — you can choose the approach that fits best.

Option A — REST API (cURL / HTTP)

Every OpenStack service exposes a standard REST API. You can interact with it directly using curl or any HTTP client. This is the lowest-level approach and gives you full control over every request. 

# 1. Authenticate and obtain a token
TOKEN=$(curl -si -X POST "$OS_AUTH_URL/auth/tokens" \
  -H "Content-Type: application/json" \
  -d '{
    "auth": {
      "identity": {
        "methods": ["application_credential"],
        "application_credential": {
          "id": "'"$OS_APPLICATION_CREDENTIAL_ID"'",
          "secret": "'"$OS_APPLICATION_CREDENTIAL_SECRET"'"
        }
      }
    }
  }' | grep -i "X-Subject-Token" | awk '{print $2}' | tr -d '\r')

# 2. List servers using the Compute API
curl -s -H "X-Auth-Token: $TOKEN" \
  "$NOVA_ENDPOINT/servers" | python3 -m json.tool

Option B — OpenStack CLI

The openstack command-line client wraps the REST APIs into simple, human-friendly commands. It is the recommended approach for day-to-day operations and is used throughout this training. 

# Install the CLI
pip3 install python-openstackclient

# Examples
openstack server list
openstack network create my-network
openstack volume create --size 10 my-volume

Option C — OpenStack SDK (Python)

The openstacksdk Python library provides a high-level, Pythonic interface to all OpenStack services. It is ideal for writing custom scripts and automation. 

# Install the SDK
pip3 install openstacksdk
import openstack

# Connect using environment variables (OS_*)
conn = openstack.connect()

# Create a network
network = conn.network.create_network(name="my-network")
print(f"Network created: {network.id}")

# Create a server
server = conn.compute.create_server(
    name="my-instance",
    flavor_id="s1-2",
    image_id="...",
    networks=[{"uuid": network.id}],
)
conn.compute.wait_for_server(server)
print(f"Server active: {server.name}")

Option D — Terraform

Terraform lets you describe your entire infrastructure as code in declarative .tf files. It handles dependency ordering, state tracking, and plan/apply workflows. 

# Initialize and apply a Terraform configuration
terraform init
terraform plan
terraform apply

Example main.tf:

# Configure the OpenStack provider
terraform {
  required_providers {
    openstack = {
      source = "terraform-provider-openstack/openstack"
    }
  }
}

resource "openstack_networking_network_v2" "my_network" {
  name           = "my-network"
  admin_state_up = true
}

resource "openstack_compute_instance_v2" "my_instance" {
  name        = "my-instance"
  flavor_name = "s1-2"
  image_name  = "Ubuntu 24.04"

  network {
    uuid = openstack_networking_network_v2.my_network.id
  }
}

Option E — Ansible

Ansible uses YAML playbooks to automate OpenStack operations. It is well suited for configuration management and orchestrating multi-step deployments. 

# Install the Ansible OpenStack collection
ansible-galaxy collection install openstack.cloud

Example playbook:

- name: Provision OpenStack resources
  hosts: localhost
  tasks:
    - name: Create a network
      openstack.cloud.network:
        name: my-network
        state: present

    - name: Create an instance
      openstack.cloud.server:
        name: my-instance
        flavor: s1-2
        image: "Ubuntu 24.04"
        network: my-network
        state: present
        wait: true

3. Which Approach to Choose?

Approach Best for Learning curve
REST API (cURL) Debugging, understanding the protocol High
OpenStack CLI Day-to-day operations, quick tasks Low
OpenStack SDK Custom Python scripts, automation Medium
Terraform Infrastructure as Code, reproducible environments Medium
Ansible Configuration management, orchestration Medium

In this training we primarily use the OpenStack CLI for its simplicity, but feel free to reproduce the exercises with any of the approaches above.

Summary

  • OpenStack is composed of independent services (Nova, Neutron, Cinder, Keystone, Glance).
  • You can access the infrastructure via the REST API, the OpenStack CLI, the Python SDK, Terraform, or Ansible.
  • The CLI is the simplest starting point; Terraform and Ansible are better suited for infrastructure as code.
  • All approaches authenticate using the same OS_* environment variables or application credentials.
